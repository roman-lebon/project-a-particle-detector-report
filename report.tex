\begin{abstract}
This project developed a simplified C++ simulation of a CMS-like particle detector. The program models particles passing through tracker, calorimeter and muon-chamber components, comparing their true energy with the measured detector response. The design uses two main inheritance hierarchies: one for particles and one for sub-detectors, with abstract base classes, virtual functions and polymorphism used to process different derived objects through a common interface. Smart pointers, file input, exceptions, \texttt{std::map} and random Gaussian noise were used to make the simulation safer and more configurable. Additional functionality included particle inference from detector-response patterns and missing-energy calculation for invisible particles. In the final test sample, all 16 particles from the input file were processed, no particles were classified as unknown, and the calculated missing energy was 15 MeV, matching the total energy of the two neutrinos in the file.
\end{abstract}

\section{Introduction}

This project implements a simplified particle detector simulation in C++, inspired by the Compact Muon Solenoid (CMS) experiment at the Large Hadron Collider (LHC). CMS is a general-purpose detector at one of the LHC collision points, designed to identify stable particles produced in collisions and measure their momenta and energies so that the collision can be reconstructed for further analysis \cite{cmsdetector}. The aim of this project was not to reproduce the full complexity of CMS, but to build a smaller object-oriented model that captures the same basic idea: particles pass through several detector layers and produce different patterns of response. In the simulation, charged particles can be measured by a tracker, particles that deposit energy can be measured by a calorimeter, and muons can pass through to muon chambers. Neutral weakly interacting particles, such as neutrinos, are not directly detected in this simplified model and instead contribute to missing energy. The program therefore compares each particle's true energy with the energy measured by each detector component, then uses the pattern of measurements to infer the likely particle type.

The code is built around two main class hierarchies: one for particles and one for detector components. The particle hierarchy stores common physical quantities such as four-momentum, spin and rest mass, while derived classes represent specific types such as electrons, muons, taus, neutrinos, photons and hadrons. The detector hierarchy represents sub-detectors --- the tracker, calorimeter and muon chamber --- each implementing its own detection rule.

This structure was chosen because the problem naturally involves shared properties alongside type-specific data: different particle types need different additional data, such as lepton number, charge, lifetime or flavour. Similarly, all sub-detectors can attempt to measure a particle, but each responds differently. Abstract base classes and virtual functions therefore allow the main detector code to treat particles and sub-detectors generically, while still calling the correct derived-class behaviour at run time.

The final program reads particles from an input text file, constructs a CMS-like detector from sub-detector components, passes each particle through it, prints the true and measured energies, infers the likely particle type, and calculates missing energy from invisible particles. Additional functionality includes a realistic calorimeter mode with Gaussian noise and exception handling for invalid input.

\section{Code Design and Implementation}

\subsection{Class Hierarchies}

The design uses two main inheritance hierarchies, shown in Figs.~\ref{fig:particleuml} and \ref{fig:detectoruml}. The first hierarchy represents particles. The abstract \texttt{Particle} class stores the quantities shared by all particles, including four-momentum, spin, antiparticle status and rest mass. More specific particle classes then add only the data that they need: \texttt{Lepton} stores lepton number and charge, \texttt{Tau} adds a lifetime, \texttt{Neutrino} adds a flavour, and \texttt{Photon} and \texttt{Hadron} derive directly from \texttt{Particle}. This avoids repeating the same physical data in every particle class.

\begin{figure}[h]
    \centering
    \includegraphics[width=1.05\linewidth]{Particle.png}
    \caption{UML class diagram for the particle hierarchy. \texttt{Particle} is the abstract base class, with \texttt{Lepton}, \texttt{Photon} and \texttt{Hadron} as derived classes. \texttt{FourMomentum} is shown as an owned object of \texttt{Particle}.}
    \label{fig:particleuml}
\end{figure}

The second hierarchy represents the detector layers, shown in Fig.~\ref{fig:detectoruml}. The abstract \texttt{SubDetector} class provides a common interface for any detector component, while \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} implement different physical responses. This matches the structure of the problem: the program should be able to store and use detector components generically, but the measurement rule depends on which detector component is actually being used.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.95\linewidth]{Detector.png}
    \caption{UML class diagram for the detector hierarchy. \texttt{SubDetector} is the abstract base class, with \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} as derived classes. \texttt{Detector} is shown as a standalone container class.}
    \label{fig:detectoruml}
\end{figure}

The base classes are abstract because a generic particle or generic sub-detector is not physically useful on its own. A particle must be a more specific type, and a detector component must define how it measures particles. The pure virtual functions force this rule in the code.

\begin{lstlisting}[caption={Pure virtual functions from \texttt{Particle.h} and \texttt{SubDetector.h}.}, label={lst:abstractinterfaces}]
virtual void print() const = 0; // Particle.h
virtual double detect(const Particle& particle) const = 0; // SubDetector.h
\end{lstlisting}

The same interface is then used throughout the program. For example, \texttt{Detector} can process any \texttt{Particle} through any stored \texttt{SubDetector} without needing separate loops for electrons, muons, photons or hadrons. The common base type keeps the controlling code simple, while virtual functions preserve the specialised behaviour of each derived class.

\subsection{Key Design Decisions}

A key design decision was to refactor \texttt{FourMomentum}. Earlier versions used a dynamically allocated vector, but this was unnecessary because a four-momentum always has exactly four components. The final version uses \texttt{std::array<double, 4>}, which makes the fixed size explicit and avoids extra pointer-following just to access four numbers. This also made \texttt{FourMomentum} itself trivially copyable, since a fixed-size array needs no custom copy logic.

\begin{lstlisting}[caption={Fixed-size storage inside the \texttt{FourMomentum} class.}, label={lst:fourmomentumarray}]
class FourMomentum {
private:
    std::array<double, 4> components; // [E, p_x, p_y, p_z]
};
\end{lstlisting}

Each \texttt{Particle} owns its \texttt{FourMomentum} through a \texttt{std::unique\_ptr}, so the particle controls the lifetime of its four-momentum object. Because a \texttt{unique\_ptr} member is not copyable, \texttt{Particle} is not copyable by default, so it implements the full rule of five explicitly. The copy operations deep-copy the four-momentum into a new \texttt{unique\_ptr} so that copied particles do not share momentum state, while the move operations transfer the pointer instead.

\begin{lstlisting}[caption={Particle ownership of its \texttt{FourMomentum} object.}, label={lst:particlefourmomentum}]
class Particle {
private:
    std::unique_ptr<FourMomentum> four_momentum;
    double spin;
    bool is_antiparticle;

protected:
    double rest_mass;
};
\end{lstlisting}

Another important design decision was to move detector-response logic out of the particle classes. In the earlier assignment code, particles stored a hard-coded detector response string. That was useful before there were real detector classes, but it became redundant once \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} were implemented. In the final design, particles store physical properties, while each sub-detector decides what it can measure.

\begin{lstlisting}[caption={Detector-specific response logic in \texttt{Tracker::detect}.}, label={lst:trackerdetect}]
double Tracker::detect(const Particle& particle) const {
    const Lepton* lepton = dynamic_cast<const Lepton*>(&particle);
    if (lepton != nullptr && lepton->get_charge() != 0) {
        return particle.get_E();
    }

    const Hadron* hadron = dynamic_cast<const Hadron*>(&particle);
    if (hadron != nullptr && hadron->get_charge() != 0) {
        return particle.get_E();
    }

    return 0.0;
}
\end{lstlisting}

The tracker uses \texttt{dynamic\_cast} to check whether the base-class \texttt{Particle} reference is actually a \texttt{Lepton} or \texttt{Hadron} with non-zero charge. An alternative would have been to add a virtual \texttt{is\_charged()} function to \texttt{Particle}, but that would require modifying every particle class. The runtime check is less elegant, but it keeps the change local to the detector classes and is used consistently across the sub-detector hierarchy.

The calorimeter was also designed with both a perfect and a realistic mode. The perfect mode is useful for testing because it returns deterministic measurements. The realistic mode adds Gaussian noise to the measurement, which was included because the project brief suggested using random numbers to make the detector response less idealised. This keeps both behaviours inside the same \texttt{Calorimeter} class, controlled by the \texttt{realistic} data member.

\begin{lstlisting}[caption={Perfect and realistic calorimeter response in \texttt{Calorimeter::detect}.}, label={lst:calorimeterdetect}]
double Calorimeter::detect(const Particle& particle) const {
    // Neutrinos - invisible
    if (dynamic_cast<const Neutrino*>(&particle) != nullptr) {
        return 0.0;
    }

    // Determine the base measurement
    double measured = 0.0;
    if (dynamic_cast<const Muon*>(&particle) != nullptr) {
        measured = 2.0;
    } else {
        measured = particle.get_E();
    }

    // In realistic mode, add Gaussian measurement noise
    if (realistic) {
        measured += noise(generator);
    }

    return measured;
}
\end{lstlisting}

The random-number generator and normal distribution are members of \texttt{Calorimeter}, rather than local variables inside \texttt{detect()}, so the random state is preserved between measurements.

A further design decision was to perform particle inference in the \texttt{Detector} class, rather than asking each particle to identify itself. Each sub-detector reports a measurement, and the detector combines the pattern of measurements into a likely particle type. This follows the working idea that particles have different detector ``fingerprints'': for example, a muon gives a signal in the tracker, calorimeter and muon chamber, while a neutrino gives no signal.

\begin{lstlisting}[caption={Rule-based particle inference from detector measurements.}, label={lst:inferparticle}]
std::string Detector::infer_particle(const std::map<std::string, double>& measurements) {
    // Look up each sub-detector reading (defaults to 0 if the detector isnt in the map)
    double tracker = measurements.count("InnerTracker") ? measurements.at("InnerTracker") : 0.0;
    double ecal = measurements.count("ECAL") ? measurements.at("ECAL") : 0.0;
    double muon_chamber = measurements.count("OuterMuonChamber") ? measurements.at("OuterMuonChamber") : 0.0;

    // Muon: hits all three layers
    if (tracker > 0 && ecal > 0 && muon_chamber > 0) {
        return "muon";
    }

    // Electron or charged hadron: tracker + calorimeter, no muon chamber hit
    if (tracker > 0 && ecal > 0 && muon_chamber == 0) {
        return "electron or charged hadron";
    }

    // Photon or neutral hadron: calorimeter only
    if (tracker == 0 && ecal > 0 && muon_chamber == 0) {
        return "photon or neutral hadron";
    }

    // Neutrino: invisible, nothing hit
    if (tracker == 0 && ecal == 0 && muon_chamber == 0) {
        return "neutrino (invisible)";
    }

    return "unknown";
}
\end{lstlisting}

This function is \texttt{static} because the inference depends only on the measurement map passed into it, not on the internal state of a particular \texttt{Detector} object. The combined categories it returns (``electron or charged hadron'', ``photon or neutral hadron'') reflect a limitation of the single-calorimeter model, discussed in Section~4.

The missing-energy calculation was another important design decision. The aim was to compare the total true energy entering the detector with the total energy measured by the detector, so that invisible particles such as neutrinos appear as missing energy. My first approach would have been to sum every sub-detector reading for each particle, but that double-counts particles seen by more than one layer. For example, a charged hadron can be measured by both the tracker and calorimeter, even though it only has one true energy.

\begin{lstlisting}[caption={Using the largest single detector reading to avoid double-counting.}, label={lst:missingenergy}]
double Detector::calculate_missing_energy(const std::vector<std::shared_ptr<Particle>>& particles) const {
    double total_true_energy = 0.0;
    double total_measured_energy = 0.0;

    for (const auto& p : particles) {
        total_true_energy += p->get_E();

        // The detector's measurement of this particle is its best single reading
        std::map<std::string, double> results = process(*p);
        double best_reading = 0.0;
        for (const auto& result : results) {
            if (result.second > best_reading) {
                best_reading = result.second;
            }
        }
        total_measured_energy += best_reading;
    }

    return total_true_energy - total_measured_energy;
}
\end{lstlisting}

The final version therefore uses the largest single detector reading as the detector's best estimate of that particle's measured energy. This matches the simplification used elsewhere in the code: when a sub-detector detects a particle, it reports a whole-particle energy estimate rather than a partial energy deposit. Taking the maximum avoids double-counting and gives a physically sensible result.

\subsection{Advanced C++ Features}

Smart pointers are used to make ownership explicit. The \texttt{Detector} stores its sub-detectors using \texttt{std::unique\_ptr}, because each \texttt{Tracker}, \texttt{Calorimeter} or \texttt{MuonChamber} has one clear owner: the detector that contains it. Processed particles, in contrast, are stored using \texttt{std::shared\_ptr}, because the same particle object is created outside the detector and also recorded inside the detector's processed-particle history.

\begin{lstlisting}[caption={Different ownership models in the \texttt{Detector} class.}, label={lst:smartpointers}]
class Detector {
private:
    std::string name;
    std::vector<std::unique_ptr<SubDetector>> sub_detectors;
    std::vector<std::shared_ptr<Particle>> processed_particles;
};
\end{lstlisting}

The same ownership idea appears when sub-detectors are added to a detector. The \texttt{add\_sub\_detector} function takes a \texttt{std::unique\_ptr} and then uses \texttt{std::move} to transfer ownership into the detector's internal vector. This makes the ownership transfer explicit: after a sub-detector has been added, it belongs to the detector container.

\begin{lstlisting}[caption={Transferring ownership of a sub-detector into the detector.}, label={lst:addsubdetector}]
void Detector::add_sub_detector(std::unique_ptr<SubDetector> sub_detector) {
    sub_detectors.push_back(std::move(sub_detector));
}
\end{lstlisting}

Polymorphism is used when a particle is processed through the detector. The \texttt{Detector::process} function loops over the stored sub-detectors through base-class pointers. It does not need to know whether a component is actually a \texttt{Tracker}, \texttt{Calorimeter} or \texttt{MuonChamber}; it simply calls the virtual \texttt{detect} function. C++ then selects the correct derived-class implementation at run time. This is the main reason that the \texttt{SubDetector} base class is useful in the detector container.

\begin{lstlisting}[caption={Processing a particle through polymorphic sub-detectors.}, label={lst:processmap}]
std::map<std::string, double> Detector::process(const Particle& particle) const {
    std::map<std::string, double> results;
    for (const auto& sd : sub_detectors) {
        results[sd->get_name()] = sd->detect(particle);
    }
    return results;
}
\end{lstlisting}

The return type is a \texttt{std::map<std::string, double>}, keyed by sub-detector name. This lets later functions, such as particle inference, look up readings by name rather than depending on position in a vector, which would make the meaning of each entry less clear.

File input is handled by \texttt{ParticleReader}, so the set of simulated particles can be changed by editing \texttt{particles.txt} rather than changing and recompiling \texttt{main.cpp}. The reader opens the file, reads it line by line, uses the first token as the particle tag, and then constructs the corresponding derived particle object. This keeps \texttt{main()} shorter and separates input parsing from the detector simulation itself.

\begin{lstlisting}[caption={Opening the particle input file and reporting failure.}, label={lst:filereader}]
std::ifstream file(filename);
if (!file) {
    throw std::invalid_argument("ParticleReader: could not open file " + filename);
}
\end{lstlisting}

Exceptions are used for invalid input rather than silently continuing or calling \texttt{exit(1)}. This was a deliberate change because the particle and detector classes should behave like reusable library code: if something goes wrong, the calling code should be able to catch the exception and decide how to respond. The same approach is used for malformed particle lines and unknown particle tags.

\begin{lstlisting}[caption={Rejecting an unknown particle tag from the input file.}, label={lst:unknowntag}]
else {
    throw std::invalid_argument("ParticleReader: unknown particle tag - " + tag);
}
\end{lstlisting}

Together, these features make the program safer and more extensible: smart pointers express ownership, virtual functions provide a common interface across sub-detectors, \texttt{std::map} keeps readings organised by name, and exceptions make invalid input fail clearly.

\section{Results}

The program was tested using the input file \texttt{particles.txt}. This file contains a mixture of charged leptons, neutrinos, taus, photons and hadrons, so that each part of the detector response is exercised. A short extract of the input format is shown in Listing~\ref{lst:particleinput}. The first word on each line gives the particle type, followed by its energy and momentum components, with extra fields where needed, such as neutrino flavour or hadron type.

\begin{lstlisting}[caption={Extract from \texttt{particles.txt}.}, label={lst:particleinput}]
ELECTRON 0.511 0.1 0.2 0.0 0
MUON 105.66 1.0 2.0 3.0 0
NEUTRINO 10.0 5.0 3.0 1.0 0 muon
PHOTON 60.0 20.0 30.0 10.0
HADRON 938.27 100.0 50.0 30.0 1.0 938.27 0 proton
\end{lstlisting}

In \texttt{main()}, these particles are passed through a simple CMS-like detector containing an inner tracker, an electromagnetic calorimeter and an outer muon chamber. The output shows both the true particle energy and the measured response from each detector component. For example, a muon leaves a tracker signal, deposits a small fixed amount of energy in the calorimeter, and is also detected by the muon chamber:

\begin{lstlisting}[caption={Example detector output for a muon.}, label={lst:muonoutput}]
Four-momentum (E, p_x, p_y, p_z) = (105.66, 1, 2, 3) MeV

Measurements:
  ECAL: 2 MeV
  InnerTracker: 105.66 MeV
  OuterMuonChamber: 105.66 MeV
Inferred particle: muon
\end{lstlisting}

This is the expected pattern: the muon is charged, so it is seen in the tracker, and it also reaches the muon chamber. The smaller calorimeter response reflects the treatment of muons as minimum-ionising particles rather than ones that deposit their full energy.

The photon output gives a different pattern: no tracker or muon chamber signal, but it is measured by the calorimeter:

\begin{lstlisting}[caption={Example detector output for a photon.}, label={lst:photonoutput}]
Four-momentum (E, p_x, p_y, p_z) = (60, 20, 30, 10) MeV

Measurements:
  ECAL: 60 MeV
  InnerTracker: 0 MeV
  OuterMuonChamber: 0 MeV
Inferred particle: photon or neutral hadron
\end{lstlisting}

The inference result is deliberately not just ``photon'', because in this detector model a neutral hadron can give the same pattern: calorimeter energy without a track or muon-chamber hit. This shows both that the inference logic is working and that the simplified detector has limited particle-identification power.

The missing-energy result provides a useful check on the implementation. The input file contains two neutrinos with energies 10 MeV and 5 MeV. Since neutrinos are invisible to all three sub-detectors in this simulation, the program returns:

\begin{lstlisting}[caption={Missing-energy output.}, label={lst:missingenergyoutput}]
[Missing energy]
Total missing energy = 15 MeV
(energy carried away invisibly, mostly by neutrinos)
\end{lstlisting}

This is a clean validation result because the calculated missing energy matches the total true energy of the two neutrinos exactly. The final summary also shows that all 16 particles from the input file were processed and classified, with no unknown cases in this test sample.

\section{Discussion and Conclusions}

The program meets the main aim of the project: it builds a simplified particle-detector simulation using object-oriented C++. The particle hierarchy and sub-detector hierarchy make the design close to the physical system being modelled, while the generic \texttt{Detector} class allows different detector components to be stored and used through a common interface. The output shows that different particle types produce different detector-response patterns, and the missing-energy result gives a simple numerical check that invisible particles are being handled correctly.

The main limitation is that the detector model is still very simplified. The code has one calorimeter class called \texttt{Calorimeter}, rather than separate electromagnetic and hadronic calorimeters. Because of this, the inference logic cannot cleanly distinguish electrons from charged hadrons, or photons from neutral hadrons. A concrete extension would be to derive \texttt{ECAL} and \texttt{HCAL} classes from \texttt{SubDetector}, or from a more specific calorimeter base class. Then electrons and photons could be made to deposit energy mainly in the ECAL, while hadrons could have a different HCAL response.

A second limitation is the tau model. The \texttt{Tau} class stores a lifetime, but the code does not simulate tau decays, so charged taus behave like other charged leptons and are inferred as ``electron or charged hadron''. A more realistic extension would add decay products, allowing a tau to produce lighter leptons, hadrons and neutrinos before the detector response is calculated.

The realistic calorimeter mode is also deliberately simple. It adds Gaussian noise to the measured energy, which is useful for demonstrating random detector smearing, but it can give unphysical negative readings for very low-energy particles. A direct improvement would be to clamp negative measurements to zero or to make the resolution depend on energy, rather than using one fixed noise distribution for every particle.

Finally, the inference function currently depends on hard-coded detector names such as \texttt{InnerTracker}, \texttt{ECAL} and \texttt{OuterMuonChamber}. This works for the present detector, but it would be less flexible if more sub-detectors were added. A better design would give each sub-detector a type or role, so the inference code could search for tracker, calorimeter and muon responses without depending on exact strings. Overall, extending the simulation is straightforward. Because the design depends on the abstract \texttt{Particle} and \texttt{SubDetector} interfaces rather than concrete types, each of the improvements above can be made by deriving a new class from an existing abstract base, with no change to the \texttt{Detector} processing loop.
