\section{Introduction}

This project implements a simplified particle detector simulation in C++, inspired by the Compact Muon Solenoid (CMS) experiment at the Large Hadron Collider (LHS). CMS is a general-purpose detector at one of the LHC collision points, designed to identify stable particles produced in collisions and measure their momenta and energies so that the collision can be reconstructed for further analysis \cite{cmsdetector}. The aim of this project was not to reproduce the full complexity of CMS, but to build a smaller object-oriented model that captures the same basic idea: particles pass through several detector layers and produce different patterns of response. The physical model used in the program is based on a CMS-like detector made from sub-detector components. In the simulation, charged particles can be measured by a tracker, particles that deposit energy can be measured by a calorimeter, and muons can pass through to muon chambers. Neutral weakly interacting particles, such as neutrinos, are not directly detected in this simplified model and instead contribute to missing energy. The program therefore compares each particle's true energy with the energy measured by each detector component, then uses the pattern of measurements to infer the likely particle type.

The programming aim was to represent this physical situation using an object-oriented C++ design. The code is built around two main class hierarchies: one for particles and one for detector components. The particle hierarchy stores common physical quantities such as four-momentum, spin and rest mass, while derived classes represent specific particle types such as electrons, muons, taus, neutrinos, photons and hadrons. The detector hierarchy represents sub-detectors such as the tracker, calorimeter and muon chamber, each of which implements its own detection rule.

This structure was chosen because the problem naturally involves both shared properties, but different particle types need different additional data, such as lepton number, charge, lifetime or flavour. Similarly, all sub-detectors can attempt to measure a particle, but each sub-detector responds differently. Abstract base classes and virtual functions therefore allow the main detector code to treat particles and sub-detectors generically, while still calling the correct derived-class behaviour at run time.

The final program reads particles from an input text file, constructs a CMS-like detector from sub-detector components, passes each particle through the detector, prints the true and measured energies, infers the likely particle type from the detector signals, and calculates missing energy from invisible particles. Additional functionality includes a realistic calorimeter mode with Gaussian noise and exception handling for invalid particle input.

\section{Code Design and Implementation}

\subsection{Class Hierarchies}

The design uses two main inheritance hierarchies, shown in Figs.~\ref{fig:particleuml} and \ref{fig:detectoruml}. The first hierarchy represents particles. The abstract \texttt{Particle} class stores the quantities shared by all particles, including four-momentum, spin, antiparticle status and rest mass. More specific particle classes then add only the data that they need: \texttt{Lepton} stores lepton number and charge, \texttt{Tau} adds a lifetime, \texttt{Neutrino} adds a flavour, and \texttt{Photon} and \texttt{Hadron} derive directly from \texttt{Particle}. This avoids repeating the same physical data in every particle class.

\begin{figure}[h]
    \centering
    \includegraphics[width=1.05\linewidth]{Particle.png}
    \caption{UML class diagram for the particle hierarchy. \texttt{Particle} is the abstract base class, with \texttt{Lepton}, \texttt{Photon} and \texttt{Hadron} as derived classes. \texttt{FourMomentum} is shown as an owned object of \texttt{Particle}.}
    \label{fig:particleuml}
\end{figure}

The second hierarchy represents the detector layers, shown in Fig.~\ref{fig:detectoruml}. The abstract \texttt{SubDetector} class provides a common interface for any detector component, while \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} implement different physical responses. This matches the structure of the problem: the program should be able to store and use detector components generically, but the measurement rule depends on which detector component is actually being used.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.95\linewidth]{Detector.png}
    \caption{UML class diagram for the detector hierarchy. \texttt{SubDetector} is the abstract base class, with \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} as derived classes. \texttt{Detector} is shown as a standalone container class.}
    \label{fig:detectoruml}
\end{figure}

The base classes are abstract because a generic particle or generic sub-detector is not physically useful on its own. A particle must be a more specific type, and a detector component must define how it measures particles. The pure virtual functions force this rule in the code.

\begin{lstlisting}[caption={Pure virtual functions from \texttt{Particle.h} and \texttt{SubDetector.h}.}, label={lst:abstractinterfaces}]
virtual void print() const = 0; // Particle.h
virtual double detect(const Particle& particle) const = 0; // SubDetector.h
\end{lstlisting}

The same interface is then used throughout the program. For example, \texttt{Detector} can process any \texttt{Particle} through any stored \texttt{SubDetector} without needing separate loops for electrons, muons, photons or hadrons. This is the main reason the hierarchy is useful: the common base type keeps the controlling code simple, while virtual functions preserve the specialised behaviour of each derived class.

\subsection{Key Design Decisions}

A key design decision was to refactor \texttt{FourMomentum}. Earlier versionsused a dynamically allocated vector, but this was unnecessary because a four-momentum always has exactly four components. The final version uses \texttt{std::array<double, 4>}, which makes the fixed size explicit and avoids extra pointer-following just to access four numbers. This also made \texttt{FourMomentum} simpler to copy, instead of needing rule-of-five code to work around ownership of a dynamically allocated container.

\begin{lstlisting}[caption={Fixed-size storage inside the \texttt{FourMomentum} class.}, label={lst:fourmomentumarray}]
class FourMomentum {
private:
    std::array<double, 4> components; // [E, p_x, p_y, p_z]
};
\end{lstlisting}
Each \texttt{Particle} still owns its \texttt{FourMomentum} through a \texttt{std::unique\_ptr}, so the particle controls the lifetime of its four-momentum object. However, the four numerical values inside\texttt{FourMomentum} are stored directly in the object using a fixed-size array. This separates object ownership from the internal representation of the physical data.

\begin{lstlisting}[caption={Particle ownership of its \texttt{FourMomentum} object.}, label={lst:particlefourmomentum}]
class Particle {
private:
    std::unique_ptr<FourMomentum> four_momentum;
    double spin;
    bool is_antiparticle;

protected:
    double rest_mass;
};
\end{lstlisting}

Another important design decision was to move detector-response logic out of the particle classes. In the earlier assignment code, particles stored a hard-coded detector response string. That was useful before there were real detector classes, but it became redundant once \texttt{Tracker}, \texttt{Calorimeter} and \texttt{MuonChamber} were implemented. In the final design, particles store physical properties, while each sub-detector decides what it can measure.

\begin{lstlisting}[caption={Detector-specific response logic in \texttt{Tracker::detect}.}, label={lst:trackerdetect}]
double Tracker::detect(const Particle& particle) const {
    const Lepton* lepton = dynamic_cast<const Lepton*>(&particle);
    if (lepton != nullptr && lepton->get_charge() != 0) {
        return particle.get_E();
    }

    const Hadron* hadron = dynamic_cast<const Hadron*>(&particle);
    if (hadron != nullptr && hadron->get_charge() != 0) {
        return particle.get_E();
    }

    return 0.0;
}
\end{lstlisting}

The tracker uses \texttt{dynamic\_cast} to check whether the base-class \texttt{Particle} reference is actually a \texttt{Lepton} or \texttt{Hadron} with non-zero charge. An alternative would have been to add a virtual \texttt{is\_charged()} function to \texttt{Particle}, but that would require modifying every particle class. The runtime check is less elegant, but it keeps the change local to the detector classes and is used consistently across the sub-detector hierarchy.
